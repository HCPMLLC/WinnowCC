# PROMPT7: Interview Probability Formula — Full Implementation## PurposeImplement the benchmark-based Interview Probability model fully and exactly across the Winnow stack. The formula quantifies interview likelihood using recruiter-aligned weights and a referral multiplier. Implement in stages; preserve explainability and compliance (heuristic, not guaranteed probability).---## Triggers — When to Use This Prompt- Implementing or refining the Interview Probability / Interview Readiness score.- Adding cover letter scoring, application logistics, or referral multiplier to matching.- Aligning match scoring with recruiter benchmarks (70/20/10, referral 8x).- Spec or product asks for "Interview Probability" or "P_i" style scoring.---## The Interview Probability Formula**Formula:**  \( P_i = \big[(W_r \cdot R_s) + (W_c \cdot C_s) + (W_a \cdot A_s)\big] \times M_{net} \)**Weights (recruiter-aligned):**- \( W_r \) (Resume): **0.70** — primary screening tool- \( W_c \) (Cover Letter): **0.20** — secondary tie-breaker- \( W_a \) (Application Logistics): **0.10** — timing and platform**Component scores (each 0–100):**- **R_s** — Resume Score: keyword/match to job + evidence strength (e.g. XYZ-style bullets).- **C_s** — Cover Letter Score: quality/relevance of cover letter for the job.- **A_s** — Application Logistics Score: timing (e.g. apply in first 10 days) + platform (e.g. Indeed vs LinkedIn) if available.**Networking multiplier:**- **M_net** = 1.0 (cold apply) or 8.0 (referred/sourced by recruiter).**Output:** Cap or normalize \( P_i \) for display (e.g. 0–100 index or 0–1). Present as "benchmark-based heuristic" in UI and legal; do not claim guaranteed probability.---## Stage 1: Resume Score (R_s)**Trigger:** Start here; R_s is the core and can replace or refine the current "match score" + "interview readiness" inputs.**Steps:**1. **Define R_s (0–100) from existing matching**   - In `services/api/app/services/matching.py`, compute a **Resume Score** that combines:     - Keyword/skill match to job description (existing skill overlap logic).     - Evidence strength: count of profile bullets that support matched skills (existing `_evidence_strength`).     - Gap penalty: missing must-have skills (existing `missing_skills` / gaps_severity).   - Optionally refine evidence to favor **XYZ-style bullets** (Accomplished [X] as measured by [Y], by doing [Z]) in parsing or in a small sub-score (e.g. bonus for bullets with numbers/outcomes).   - Output a single R_s in 0–100 (e.g. same weighted blend as current readiness, or 0.6×match + 0.2×evidence + 0.2×(20−gaps) then scaled to 100).   - Preserve **reasons** (matched_skills, missing_skills, evidence_refs) for explainability in API and UI.2. **API and DB**   - Add or repurpose a field for **resume_score** (R_s) in the Match model and in match response schemas (`app/schemas/matches.py`, `app/routers/matches.py`).   - If you keep existing `match_score` and `interview_readiness_score` for backward compatibility, document that R_s is the new canonical resume component for P_i; otherwise replace them with R_s in the scoring pipeline and responses.3. **Frontend**   - In `apps/web/app/matches/page.tsx` (and any MJASS/draft explain UIs), show R_s as "Resume Score" or fold it into the single "Interview Probability" display once P_i is implemented. Ensure tooltips or copy state it is heuristic.**Done when:** R_s is computed, stored, returned by the API, and visible (or wired) in the UI.---## Stage 2: Application Logistics Score (A_s)**Trigger:** After R_s is in place; adds "when" and optionally "where" you apply.**Steps:**1. **Timing (first 10 days)**   - In `matching.py`, for each job use `job.posted_at` and the match/compute time. Compute "days since posted" (e.g. 0–10+).   - Define a timing sub-score (0–100): e.g. 100 if applied (or matched) within first 10 days, linear decay after, or step function (e.g. 100 for 0–10 days, 50 for 11–14, 0 for 15+). Use "days since posted" as proxy for "apply early" if application timestamp is not yet stored.   - If you later add "application timestamp" when the user marks "Applied," use that for A_s timing instead of match time.2. **Platform (optional)**   - If job has a `source` (e.g. remotive, indeed, linkedin), map to a platform score (e.g. Indeed 80–100, LinkedIn 30–50 per benchmarks; exact numbers are configurable). If no source or "application source" exists, use a default (e.g. 50) or skip platform and weight timing 100% within A_s.   - If you add an "application source" field when user marks "Applied" (Indeed vs LinkedIn vs Other), use that for platform; else use job source as proxy.3. **Combine into A_s**   - A_s = weighted combination of timing + platform (e.g. 0.6×timing + 0.4×platform, or 1.0×timing if platform not available). Clamp to 0–100.   - Store A_s on Match (add column if needed) or compute on read; include in match response schema.4. **Migrations and API**   - Add DB column(s) if needed (e.g. `application_logistics_score`, or `applied_at`, `application_source`). Alembic migration in `services/api/alembic/versions/`.   - Expose A_s in match list and detail responses; document in OpenAPI/schema.**Done when:** A_s is computed (and optionally stored), returned by the API, and available for P_i.---## Stage 3: Referral Multiplier (M_net)**Trigger:** After R_s (and ideally A_s) so P_i can use the multiplier.**Steps:**1. **Data model**   - Add a field to represent referral for a **job application** or **match**: e.g. `referred` (boolean) or `referral_source` (string/null) on Match, or on a per-application record if you have one.   - If referral is global per user (e.g. "I have a referral for this job" at apply time), store it on the Match or application record when the user marks referred.   - Alembic migration: add `referred` (boolean, default False) and optionally `referral_source` to the Match table (or equivalent).2. **Scoring**   - In `matching.py`, when computing P_i (or in a post-step), set \( M_{net} = 8.0 \) if `referred` is true, else \( M_{net} = 1.0 \).   - Apply M_net only to the final P_i (see Stage 5). Do not apply to R_s, C_s, or A_s individually.3. **API and product**   - Add way to set `referred` (e.g. PATCH match or "Mark as referred" when saving/apply). Expose `referred` and optionally `referral_source` in match/application APIs.   - UI: when user marks "Applied" or views a match, allow "I have a referral" and persist it so M_net = 8 for that match when P_i is computed or displayed.**Done when:** M_net is 1.0 or 8.0 per match based on referral flag; API and UI support setting and showing it.---## Stage 4: Cover Letter Score (C_s)**Trigger:** After cover letter generation exists; C_s requires a cover letter per job (or placeholder).**Steps:**1. **Cover letter feature (if not present)**   - Add flow: user can generate a **job-specific cover letter** from profile + job (same pattern as tailored resume). Store per user/job (e.g. in DB or object storage); link to Match or application.   - Schema: e.g. CoverLetter (user_id, job_id, content, generated_at, version). API: create/update/get cover letter for a match/job.2. **C_s scoring (0–100)**   - Implement a **cover letter score** based on:     - Keyword alignment with job description (similar to resume keyword match).     - Optional: structure/quality heuristics (length, sections, mention of company/role).   - Inputs: cover letter text + job description (and maybe profile). Output: 0–100.   - If no cover letter exists for the match, set C_s = 0 (or a neutral 50); document behavior in spec.3. **When to compute C_s**   - On generation: when user generates a cover letter for a job, compute C_s and store it (e.g. on Match or on CoverLetter record).   - On match compute: if matches are recomputed and a cover letter exists for that job, refresh C_s; else C_s = 0 or default.   - Expose C_s in match response and in job detail (e.g. "Cover letter score: 72").4. **API and DB**   - Store C_s (e.g. `cover_letter_score` on Match or on CoverLetter). Migration if needed.   - Return C_s in match list/detail and in any "explain" payload for drafts/MJASS.**Done when:** C_s is computed for matches that have a cover letter, stored, returned by API, and available for P_i.---## Stage 5: Combine into P_i and Expose**Trigger:** After at least R_s and optionally A_s, M_net, C_s are available.**Steps:**1. **Compute P_i**   - In `matching.py` (or a dedicated scoring helper), compute:     - \( P_i = \big[(W_r \cdot R_s) + (W_c \cdot C_s) + (W_a \cdot A_s)\big] \times M_{net} \)   - Use 0 for any component not yet implemented (e.g. C_s = 0 until Stage 4, A_s = 50 or 0 until Stage 2). Weights: W_r = 0.70, W_c = 0.20, W_a = 0.10.   - **Cap or normalize for display:** e.g. cap at 100, or use a 0–100 index (e.g. min(100, P_i)) so the product never shows ">100%" as a probability. Document that the raw formula can exceed 100 when M_net = 8.2. **Storage and API**   - Store P_i on Match (e.g. `interview_probability` or keep `interview_readiness_score` as the field name and set it from P_i). Ensure one canonical field for "interview probability / readiness" in responses.   - Return P_i (and optionally R_s, C_s, A_s, M_net) in match list and match detail. Optionally return in MJASS/draft explain payload.3. **Frontend**   - Replace or rename "Interview Readiness Score" to "Interview Probability" (or keep "Interview Readiness" with tooltip: "Benchmark-based estimate"). Show P_i as primary; optionally show R_s, C_s, A_s in an "Explain" or "How we score" section.   - Copy and compliance: use phrases like "benchmark-based estimate," "heuristic," "not a guarantee." Align with SPEC.md (no guaranteed probability claims).4. **Backward compatibility**   - If existing clients expect `interview_readiness_score`, keep the field and set it to the capped P_i value so behavior is consistent.**Done when:** P_i is computed, stored, returned, and displayed; UI copy is compliant; R_s (and when implemented C_s, A_s, M_net) are available for explainability.---## Compliance and Positioning- **SPEC.md:** Keep "Interview Readiness Score (heuristic, explainable; not a guaranteed probability)" and "No guaranteed interview probability claims" in non-goals.- **UI:** Use "Interview Probability" or "Interview Readiness" with a short tooltip or help text: e.g. "Based on resume fit, cover letter, application timing, and referral—a benchmark-based estimate, not a guarantee."- **Legal/marketing:** Do not claim a literal probability of interview; position as an index or readiness estimate informed by recruiter benchmarks.---## File and Component Reference- **Scoring logic:** `services/api/app/services/matching.py`- **Match model:** `services/api/app/models/match.py`- **Match schemas:** `services/api/app/schemas/matches.py`- **Match API:** `services/api/app/routers/matches.py`- **Frontend matches:** `apps/web/app/matches/page.tsx`- **MJASS/draft explain:** `apps/web/app/mjass/review/page.tsx` (and any API returning explain)- **Migrations:** `services/api/alembic/versions/`- **Product spec:** `SPEC.md`---## Summary Checklist- [ ] Stage 1: R_s computed, stored, returned, and used in UI/explain.- [ ] Stage 2: A_s (timing + optional platform) computed and exposed.- [ ] Stage 3: M_net (referred flag) on Match, applied in P_i.- [ ] Stage 4: Cover letter generation + C_s scoring and storage.- [ ] Stage 5: P_i = [(W_r·R_s)+(W_c·C_s)+(W_a·A_s)]×M_net; capped; exposed in API and UI with compliant copy.
