## Database Schema Overview

### New Tables

#### 1. `employer_profiles`
Stores employer company information and subscription details.

**Key Fields:**
- `user_id` - Links to the user account
- `company_name`, `company_size`, `industry` - Company info
- `subscription_tier` - 'free', 'starter', 'pro', 'enterprise'
- `subscription_status` - 'active', 'cancelled', 'past_due'
- `stripe_customer_id`, `stripe_subscription_id` - Billing integration

**Subscription Tiers (Planned):**
- **Free:** 1 active job, 10 candidate views/month
- **Starter:** 5 active jobs, 50 candidate views/month ($99/mo)
- **Pro:** Unlimited jobs, 200 candidate views/month ($299/mo)
- **Enterprise:** Custom pricing, unlimited everything

---

#### 2. `employer_jobs`
Job postings created by employers.

**Key Fields:**
- `employer_id` - Links to employer profile
- `title`, `description`, `requirements` - Job details
- `remote_policy` - 'on-site', 'hybrid', 'remote'
- `employment_type` - 'full-time', 'part-time', 'contract', 'internship'
- `salary_min`, `salary_max`, `salary_currency` - Compensation
- `status` - 'draft', 'active', 'paused', 'closed'
- `view_count`, `application_count` - Analytics

---

#### 3. `employer_candidate_views`
Tracks when employers view candidate profiles (for analytics and usage limits).

**Key Fields:**
- `employer_id`, `candidate_id` - Who viewed whom
- `viewed_at` - Timestamp
- `source` - How they found the candidate ('search', 'job_application', 'recommendation')

**Purpose:** 
- Enforce subscription limits (e.g., free tier = 10 views/month)
- Analytics for candidates (who's viewing your profile)
- Track engagement metrics

---

#### 4. `employer_saved_candidates`
Employer's saved/favorited candidates.

**Key Fields:**
- `employer_id`, `candidate_id` - The relationship
- `notes` - Employer's private notes about the candidate
- `saved_at` - When they saved them

**Constraint:** Unique constraint on (employer_id, candidate_id) - can't save the same candidate twice

---

### Modified Tables

#### `users`
**Added:**
- `role` - VARCHAR(20), values: 'candidate', 'employer', 'both'

**Purpose:** Support users who are both job seekers and hiring managers

---

#### `candidate_profiles`
**Added:**
- `open_to_opportunities` - BOOLEAN, default true
- `profile_visibility` - VARCHAR(50), values: 'public', 'anonymous', 'private'

**Purpose:**
- `open_to_opportunities` - Candidates can opt in/out of employer searches
- `profile_visibility`:
  - **public** - Name and full profile visible to employers
  - **anonymous** - Profile visible but name hidden ("Candidate ABC123")
  - **private** - Not visible in employer searches at all

---

## Indexes Created

For query performance, the following indexes were created:

1. `idx_employer_jobs_status` - Fast filtering of jobs by status
2. `idx_employer_jobs_employer_id` - Fast lookup of jobs by employer
3. `idx_employer_candidate_views_employer` - Fast analytics queries
4. `idx_employer_saved_candidates_employer` - Fast retrieval of saved candidates

## Data Model Relationships

```
User (role: 'employer')
  └─> EmployerProfile (1:1)
        └─> EmployerJobs (1:many)
        └─> EmployerCandidateViews (1:many)
        └─> EmployerSavedCandidates (1:many)

User (role: 'candidate')
  └─> CandidateProfile (1:1)
        └─> (viewed by) EmployerCandidateViews (1:many)
        └─> (saved by) EmployerSavedCandidates (1:many)

User (role: 'both')
  ├─> EmployerProfile (1:1)
  └─> CandidateProfile (1:1)
```

## Troubleshooting

### Error: "relation 'users' does not exist"
**Cause:** Users table hasn't been created yet  
**Solution:** Make sure you've run all previous migrations first with `alembic upgrade head`

### Error: "column 'role' already exists"
**Cause:** Migration has already been run  
**Solution:** Check migration status with `alembic current`

### Error: "database connection failed"
**Cause:** Database isn't running or connection string is wrong  
**Solution:** 
1. Check Docker is running: `docker ps`
2. Verify database container is up
3. Check `.env` file has correct `DATABASE_URL`

### Can't find migration file
**Cause:** Not in correct directory  
**Solution:** Make sure you're in `services/api` when running alembic commands
